## Находка 1 — безопасность n8n

- Полный редактор n8n (не только вебхук) открыт напрямую на `restai.space`, без второго слоя защиты (нет IP-allowlist, нет Basic Auth перед панелью — в отличие от Supabase, где Basic Auth есть).
- Логин n8n сам по себе работает (`/rest/login` возвращает 401 без учётных данных) — не настежь, но единственный барьер.
- n8n поднят с `environment=development` (видно в мета-теге страницы, конфиг Sentry: `n8n@2.2.5`, `environment: development`), а не `production`.
- Не проверяли: хранится ли в credentials каких-то workflow ключ Supabase `service_role` — если да, это самый ценный актив на этом сервере и приоритет №1 для защиты.

### Что сделать

1. В nginx (`/etc/nginx/fastpanel2-sites/restai_space_usr/restai.space.conf`) добавить ограничение доступа (IP-allowlist или Basic Auth) на всё, КРОМЕ `location` с `/webhook/*` — вебхук обязан остаться публичным для гостевого чата, редактор — нет.
2. Проверить credentials во всех workflow n8n на предмет хранящихся там ключей Supabase/других секретов.
3. Сменить пароль администратора n8n, если давно не менялся; включить 2FA, если поддерживается версией 2.2.5.
4. Перевести `environment` на `production`, если это управляется переменной окружения (уточнить у n8n конкретно для этой версии).

## Находка 2 — производительность / нагрузка

Архитектура: n8n в режиме `regular` (один процесс Node.js, без очереди/воркеров) + SQLite для лога выполнений (файловая блокировка на запись — потолок независимо от железа) + 1 vCPU / 1.9 GB RAM, разделяемые с apache2/php-fpm/named.

- **Работает без проблем**: обычный поток гостевых сообщений — единицы одновременных диалогов с интервалом в несколько секунд. Узкого места нет, процесс большую часть времени просто ждёт ответ ИИ-бэкенда.
- **Первые звоночки**: примерно от нескольких десятков (~20-30) одновременных executions, накладывающихся друг на друга по времени записи в SQLite.
- **Реальные глюки**: устойчивая нагрузка от ~100+ запросов в минуту (особенно пачками, не равномерно) — ожидаемы ошибки "database is locked" и таймауты, которые в приложении сейчас молча превращаются в заглушку "Извините, я немного отвлёкся" (нет ретрая на клиенте).
- Для одного ресторана в обычном режиме зала — практически недостижимо. Риск реальный при масштабировании на много ресторанов одновременно.
- Оценка по архитектуре, не по реальному нагрузочному тесту — при необходимости точных цифр можно написать скрипт, параллельно бомбардирующий вебхук N запросами.

## План работ: SQLite → Postgres для n8n (дешёвый способ отодвинуть потолок)

Цель: убрать блокировку SQLite как узкое место, не переезжая на сервер Supabase и не меняя n8n на другое решение. После перехода потолок начинает определяться одним vCPU (для I/O-bound нагрузки — на порядок выше, сотни одновременных executions вместо десятков).

Важно: это не флаг, а перенос данных — при простом переключении DB_TYPE все текущие workflows/credentials "исчезнут" из вида (SQLite-файл останется на диске, но n8n перестанет туда смотреть).

### Шаги

**0. Бэкап (обязательно перед всем остальным)**

```bash
docker exec n8n_server n8n export:workflow --all --output=/tmp/workflows_backup.json
docker exec n8n_server n8n export:credentials --all --output=/tmp/creds_backup.json
docker cp n8n_server:/tmp/workflows_backup.json ./workflows_backup.json
docker cp n8n_server:/tmp/creds_backup.json ./creds_backup.json
docker cp n8n_server:/home/node/.n8n/database.sqlite ./database.sqlite.bak
```

**1. Поднять Postgres локально на сервере 1** (новый контейнер, НЕ на сервере Supabase) — добавить в тот же docker-compose, что и n8n:

```yaml
  n8n_postgres:
    image: postgres:16-alpine
    restart: always
    environment:
      POSTGRES_USER: n8n
      POSTGRES_PASSWORD: <сложный-пароль>
      POSTGRES_DB: n8n
    volumes:
      - n8n_postgres_data:/var/lib/postgresql/data
volumes:
  n8n_postgres_data:
```

**2. Переменные окружения n8n:**

```
DB_TYPE=postgresdb
DB_POSTGRESDB_HOST=n8n_postgres
DB_POSTGRESDB_PORT=5432
DB_POSTGRESDB_DATABASE=n8n
DB_POSTGRESDB_USER=n8n
DB_POSTGRESDB_PASSWORD=<тот же пароль>
```

`N8N_ENCRYPTION_KEY` — оставить ТЕМ ЖЕ, что стоит сейчас (иначе импортированные credentials не расшифруются).

**3. Перезапуск и импорт**

```bash
docker compose up -d n8n_postgres
docker compose up -d --force-recreate n8n
docker exec n8n_server n8n import:workflow --input=/tmp/workflows_backup.json
docker exec n8n_server n8n import:credentials --input=/tmp/creds_backup.json
```

**4. Проверка**

- Зайти в редактор, убедиться, что все workflow на месте и активны (после импорта активацию иногда нужно включить заново вручную)
- Прогнать реальный тестовый запрос через вебхук гостевого чата, убедиться, что ответ приходит
- Проверить логин администратора

**5. Не удалять `database.sqlite.bak` сразу** — подержать в стороне, пока не подтверждено, что всё стабильно работает на новой базе.

Старая история executions при переходе не переносится — это ожидаемо, не критично (это просто прошлые логи).

### Дополнительно

Освободить память перед добавлением Postgres: apache2/php-fpm/named на сервере 1 похоже простаивают (обслуживают тот самый осиротевший insis_pro_usr) — стоит проверить и, если не используются, отключить, чтобы дать Postgres место на тесной машине (1.9GB).

Делать миграцию в тихое окно (не в разгар работы ресторанов), т.к. это прямой путь гостевого чата.

## Отложенные / открытые вопросы

Не проверено, совпадает ли это с реальным доменом гостевого приложения RestAI в Vercel — если нет, гостевой чат может тихо падать в CORS-ошибку на реальном домене (замаскировано под общий catch → "Извините, я немного отвлёкся").
- Альтернатива/следующий шаг после Postgres-миграции: перенести гостевой AI-чат с n8n на прямой вызов DeepSeek через новую Vercel serverless-функцию (`/api/chat.js`, по аналогии с `/api/menu.js`/`/api/branding.js`) — обсуждалось, решение не принято. DeepSeek не заблокирован в РФ (в отличие от OpenAI/Anthropic), API OpenAI-совместимый.
- Осиротевший nginx-конфиг `insis_pro_usr` на сервере 1 — не убран, не критично.
