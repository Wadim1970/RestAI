# voice-relay

Сервис-ретранслятор для голосового ИИ-помощника гостя (OpenAI Realtime
API, либо xAI Grok Voice Agent — см. «Провайдеры» ниже). Держит настоящие
API-ключи — гость никогда не подключается к провайдеру напрямую.

## Почему отдельный сервис, а не часть waiter-api

`waiter-api` стоит на VPS в Cloud.ru (Россия) — OpenAI блокирует запросы
с российских IP на уровне своей инфраструктуры, и это касается не только
сайта ChatGPT, но и API в целом. `voice-relay` поэтому должен физически
стоять на сервере в Казахстане (там же, где сейчас n8n) — тогда исходящее
соединение к OpenAI идёт с казахстанского IP.

Гость (в России, в ресторане) подключается по WebSocket к этому сервису —
это обычный трафик к нашему серверу, тут никаких ограничений нет — и уже
этот сервис от своего имени открывает соединение к OpenAI.

## Архитектура (этап 1: только голос, без синхронизации с экраном)

```
Гость (браузер) --WS--> voice-relay (Казахстан) --WS--> OpenAI Realtime API
```

При открытии соединения сервис:

1. Забирает меню ресторана через уже существующий публичный
   `GET /api/menu` (кэшируется на Vercel Edge — отдельный поход в
   Supabase не нужен).
2. Забирает профиль гостя (`guests.preferences`, `guest_restaurant_stats`)
   тем же способом, что и текстовый чат в `useChatApi.js` — анонимным
   ключом; эти данные уже открыты на SELECT анониму (см. миграцию
   `20260710020000_guest_restaurant_stats.sql`). Отдельный сервис-ключ
   Supabase здесь не нужен.
3. Собирает системный промпт и открывает голосовую сессию у OpenAI.
4. Дальше просто гоняет аудио туда-обратно.

Показ блюд/корзины на экране во время разговора (function calling у
OpenAI) — следующий этап, в этой версии ещё не реализован.

## Провайдеры

`VOICE_PROVIDER` в `.env` (`openai` | `grok`) выбирает, куда сервис
открывает вторую WS-ногу — переключение общее для всего сервиса
(перезапуск pm2), без деления по сессиям. Grok Voice Agent совместим по
протоколу с OpenAI Realtime API (см. `src/realtimeProvider.js`), ключ и
баланс — в [console.x.ai](https://console.x.ai), голос задаётся строкой
(`eve`/`ara`/`rex`/`sal`/`leo`) в `GROK_VOICE` или в профиле ресторана
(`character_profile.voice.grok_voice`).

## Эндпоинты

| Метод | Путь | Назначение |
|---|---|---|
| `GET` | `/health` | Проверка живости |
| `WS` | `/voice?guestId=&restaurantId=` | Голосовая сессия гостя |

## Локальный запуск

```bash
cd voice-relay
cp .env.example .env   # заполнить значениями
npm install
npm run dev
```

Без `OPENAI_API_KEY` сервис всё равно поднимется и ответит на `/health` —
соединение с OpenAI открывается лениво, при первом подключении гостя, и
просто упадёт в лог, если ключа нет или он неверный.

## Деплой на сервер (Казахстан)

Точные команды дадим отдельно, когда посмотрим, что уже стоит на сервере
(версия Node.js, pm2, nginx). Ожидаемо — то же самое, что уже сделано для
`Waiter-app/server` (см. его README): `npm ci`, `pm2 start src/index.js
--name voice-relay`, наружу — через nginx с TLS на отдельном поддомене
(например `voice.restai.pro`).
